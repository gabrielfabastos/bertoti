# Engenharia de Software
## Atividade 1

No primeiro trecho do livro Software Engineering at Google, os autores discutem a diferença entre programação, ciência da computação e engenharia de software.

Eu entendi que programar é basicamente escrever código, enquanto ciência da computação está mais ligada à parte teórica. Já engenharia de software envolve aplicar esse conhecimento de forma estruturada para construir sistemas que funcionem no mundo real, assim como acontece em outras engenharias, como civil ou aeronáutica.

O texto também mostra que, diferente de outras engenharias, o desenvolvimento de software nem sempre seguiu padrões tão rigorosos. Porém, como o software está cada vez mais presente em áreas críticas da sociedade, é necessário adotar práticas mais organizadas, confiáveis e sustentáveis.

Na minha visão, o principal ponto desse trecho é que engenharia de software vai além de “fazer o código funcionar”. Ela envolve responsabilidade, processos e uma preocupação maior com qualidade e confiabilidade.

Comentários do professor:
Falar sobre o software ser intangível. A nossa é a única engenharia que está desenvolvendo algo intangível.

## Atividade 2

No segundo trecho, os autores ampliam o conceito de engenharia de software. Eles explicam que não se trata apenas de escrever código, mas também de pensar nas ferramentas, processos e práticas que mantêm esse código útil ao longo do tempo.

Uma frase que resume bem essa ideia é que engenharia de software é “programação integrada ao longo do tempo”. Ou seja, o código precisa ser sustentável, preparado para mudanças e fácil de manter no futuro.

O texto destaca três princípios importantes:

Tempo e Mudança: o software precisa se adaptar às mudanças ao longo da sua vida útil.

Escala e Crescimento: tanto o sistema quanto a organização podem crescer, e o software deve acompanhar esse crescimento.

Trade-offs e Custos: toda decisão técnica envolve escolhas e consequências.

Para mim, esse trecho reforça que desenvolver software não é pensar só no presente, mas no ciclo de vida completo do sistema.

Comentários do professor:
Engenharia de software é programação. Envolve isso mas envolve programação + tempo + escala + tradeoffs
Entender como meu código evolui ao longo do tempo. Atualizações. Escala é como o software cresce, para escalar é necessário dividir tudo em partes.

Exemplos de Trade-offs
1. Velocidade vs. Qualidade

Se a equipe prioriza apenas velocidade, pode entregar rápido, mas corre o risco de gerar problemas futuros e débito técnico.
Se prioriza apenas qualidade extrema, pode atrasar entregas e perder oportunidades.

É preciso encontrar um equilíbrio entre os dois.

2. Simplicidade vs. Flexibilidade

Um sistema simples é mais fácil de entender e manter.
Já um sistema muito flexível pode atender melhor a futuras mudanças, mas tende a ser mais complexo.

A decisão depende do quanto o projeto deve evoluir no futuro.

3. Custo Inicial vs. Sustentabilidade a Longo Prazo

Uma solução rápida pode reduzir o esforço inicial, mas aumentar o custo de manutenção depois.
Já investir mais tempo no começo pode tornar o sistema mais sustentável e fácil de evoluir.

Esse trade-off está diretamente ligado à ideia de pensar o software ao longo do tempo.
